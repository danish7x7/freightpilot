# ADR-0009: Agent L3 confirmation gate — opaque token bound to a server-authoritative proposal, single-use claim, and reconcile-by-read recovery

- **Status:** accepted
- **Date:** 2026-07-21
- **Phase/Layer:** Phase 2 / agent-phase L3 (global L5) — confirmation gate + booking execution (§6.3.2)
- **Deviates from master plan:** partial — IMPLEMENTS §6.3.2 (confirmations table, 10-min TTL, `actor=agent` booking). One deliberate deviation: agent-service publishes **no OpenAPI/spectral contract** for its gate endpoints, unlike the contract-first §5 / L3-gate rule that binds rates/booking (Flag 7 ruling below).

## Context
L3 fills the seam ADR-0008 left open: `create_booking` is PROPOSE-ONLY, so nothing in the tool loop can execute a booking. This is the FIRST code path where LLM-originated intent can reach the two real booking POSTs (`POST /bookings`, then `POST /bookings/{id}/confirm`) — and §6.3's load-bearing invariant says no booking executes without an explicit human click. So L3 needs a credential a human hands back (a token), a place to store what that token authorizes, single-use semantics, and a recovery story for the two-call execution when an ack is lost. agent-service had ZERO persistence before this; L3 also gives it its first database. Triggered by `docs/journal/2026-07-21.md` Session 2 carried notes and the §6.3.2 confirmation-gate design. Reviews: code-reviewer REQUEST CHANGES → APPROVE WITH NITS (2 Blocking fixed); security-reviewer PASS WITH NOTES; six architecture-guardian conditions (A–F) verified structurally.

## Options considered
1. **Token re-carries the booking params (client resends quote_id / actor on redeem)** — pro: no server-side proposal store; con: the token then authorizes "*a* booking," not "*this* booking" — params are tamperable and a token could be replayed against a different proposal. Rejected.
2. **Opaque server-side token keys a stored, server-authoritative proposal; redeem carries ONLY the token (chosen)** — pro: the client never resends booking params, so the token authorizes exactly THIS proposal and cannot be replayed against another; server is the single source of truth for what executes. Con: needs a `confirmations` table and a store — accepted (it's also where audit + single-use live).
3. **Recovery = blind re-POST, treat a 409 on confirm as failure** — pro: simplest retry; con: **confirm is NOT idempotent** (HELD→CONFIRMED legal; CONFIRMED→CONFIRMED = 409 ILLEGAL_TRANSITION, proven by booking's own `idempotency.it.test` line 95) — a second execution would 409 on an already-CONFIRMED booking and misread a *succeeded* booking as failed. Rejected in favour of reconcile-by-READ + a single-use claim.

## Decision
The confirmation gate is a **separate service from the tool loop** (Flag 5 — `runAgentTurn` stays proposal-producing; `gateService` mints/persists/executes). agent-service **owns** the `confirmations` table (§6.3.2) — NOT booking-service, which is actor-agnostic.

1. **Token binding (Condition A).** `propose()` mints an opaque 256-bit token (`crypto.randomBytes(32).base64url`, `gate/token.ts`) that keys a stored server-authoritative proposal (quote_id, shipper_ref, actor). `redeem()` carries ONLY the token; the client never resends booking params → a token authorizes THIS booking, and cannot be replayed against a different proposal.
2. **create/confirm idempotency ASYMMETRY + reconcile-by-read (Condition D — load-bearing).** create is idempotent on `Idempotency-Key = token` (first-write-wins, verified against booking-service); confirm is NOT. On confirm failure/timeout the executor **reconciles by READ** (GET the booking: CONFIRMED ⇒ treat as success, the ack was merely lost; HELD ⇒ retry; bounded 2 attempts) — never a blind re-POST that reads a 409 as failure.
3. **Single-use claim (Condition B) is MANDATORY, not belt-and-suspenders.** `claimForRedemption` is a transactional `UPDATE ... WHERE status='pending' AND expires_at > now() RETURNING`; only the winner executes. This is *required by* the confirm asymmetry: a second execution's confirm would 409 on an already-CONFIRMED booking. TTL = 10 min (§6.3.2).
4. **Token unguessability is a DO-NOW L3 requirement (Condition F), not a deferred limitation** — because auth is a Non-Goal (ADR-0005), possession of the token IS the authorization. 256-bit crypto-random; the secret is masked out of Fastify access logs (token-masking req serializer) and kept out of `booking_events` metadata — the NON-SECRET `id` (separate column) goes into logs and booking metadata instead.
5. **Executor is structurally unreachable from the tool loop (Condition C).** `gate/executor.ts` is the ONLY code issuing the two real booking POSTs and is imported ONLY by `gateService`; a static import-graph test (hardened against dynamic import/require) asserts the loop cannot reach it — preserving the ADR-0008 L2 seam.
6. **Gate endpoints are UI-INTERNAL (Flag 7 ruling).** agent-service publishes NO OpenAPI contract (unlike rates/booking): `GET`/`POST /api/v1/confirmations/:token` are consumed by the same-origin L5 chat UI, not external clients — so no spectral/contract gate for agent. Code-reviewer flagged the absence; the ruling is deliberate and recorded here so it is tracked, not forgotten.

Deciding rationale: possession of the token is the authorization, so the token must (a) name exactly one server-owned proposal, (b) be spendable exactly once, and (c) recover from a lost ack without double-executing a non-idempotent confirm. Reconcile-by-read + single-use claim make "at most one booking per confirmation" true by construction, not by retry luck.

## Consequences
- **Easier:** the gate is auditable and safe by construction — the claim marks the row `consumed` BEFORE the token is transmitted to booking-service, so any leaked token copy is already spent (security-reviewer note). Every failure outcome is now recorded (create/confirm errors throw `BookingExecutionError` and are persisted), so a consumed row never lies as `in_progress` forever except across a process crash.
- **Harder / ruled out:** agent-service now carries a database + migrations (mirrors booking-service layout; migrate-on-boot; `agent-it` CI job). Pinned `drizzle-orm ^0.45.2` — deliberately ABOVE booking-service's still-vulnerable `^0.38.3` (ADR-0005 / GHSA-gpj5-g38j-94v9) so agent does NOT inherit the debt; booking's bump stays an open follow-up.
- **Known limitations recorded (auth is a Non-Goal per ADR-0005, acceptable-for-now):** (a) tokens are not tenant-scoped; (b) the `actor` is caller-trusted, not authenticated.
- **Revisit when:** L4 adds the **crashed-winner reaper** — with Blocking-1 fixed, the SOLE remaining path to a permanently-stuck `in_progress` is process death between `claimForRedemption` and `recordExecution`; a time-based sweep keyed on `consumed_at` age owns that edge (confirmed NOT security-critical — a succeeded booking shows as `in_progress`, never a double-book). Also revisit for X-Request-Id propagation to booking calls (correlation gap), the booking-service drizzle-orm bump, and the fleet-wide Fastify/fast-uri advisory (GHSA-v2hh-gcrm-f6hx).
