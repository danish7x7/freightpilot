## Architectural framing (why this is a courier, not a feature)
- The client is the **ORCHESTRATOR** (§6.2) — the same role the agent plays in Phase 2. It is a pure **courier**: it forwards rates-owned data it was handed, asserts nothing itself, does no rate math (§2.2), and never enforces booking transitions (§2.4 — the server's 409 is authoritative).
- This is exactly why PR-A had to add `lane_id` to the rates contract first: a courier can't forward what rates didn't emit.

## Shipped (client only)
- **`client/src/api/booking.ts`** — openapi-fetch runtime client to booking-service (`VITE_BOOKING_URL` default `:8081`), `X-Request-Id` middleware mirroring booking-service's onRequest hook, `BookingApiError` over the generated `ErrorEnvelope`. Mirrors `rates.ts` — a separate per-service client, NOT a shared factory (guardian's call).
- **`client/src/api/bookingHooks.ts`** — hooks for the 6 public booking endpoints; actor hardcoded to `const USER: Actor = "user"` so the UI can only ever act as the user — no path to forge agent/system.
- **`client/src/components/BookingPanel.tsx`** — the bridge + reserve chain: forwards the `(lane_id, rate_card_id)` pair PLUS `breakdown`/`total_cents`/`currency` verbatim from the rates `QuoteResponse` into booking `POST /quotes → hold → createBooking`. Stable idempotency key per panel instance; synchronous `inFlight` ref guard against double-submit.
- **`client/src/components/BookingDetailView.tsx`** — `GET /bookings/{id}`; Confirm (only when HELD) / Cancel buttons; surfaces 409 vetoes verbatim (client never enforces the §2.4 state machine).
- **`client/src/components/EventTimeline.tsx`** — `booking_events` timeline rendering user/agent/system actor labels + a static "ready for the agent" legend (every actor is `user` in Phase 1; agent joins Phase 2). Defensive fallback for out-of-enum actors.
- **`client/e2e/booking.{spec,fixtures}.ts`** — hermetic mocked-network Playwright, fixtures typed against `booking.gen.ts` (drift breaks the build). 3 tests: happy path (reserve→confirm→cancel, asserting courier forwarding of the id pair + money, correlation/idempotency headers, `actor='user'`, server-driven timeline); idempotency-key REUSE across a retry (no double-book); a 409 confirm veto surfaced verbatim.
