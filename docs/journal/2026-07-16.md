## Decisions — ADR-0005 (guardian-mandated)
- **ADR-0005** (`docs/decisions/0005-booking-hold-level-model-option2-idempotency.md`, accepted): booking hold-level model = **Option 2** — born QUOTED, held-on-create in one transaction (two `booking_events` rows), quote CONSUMED atomically; `/confirm` = HELD→CONFIRMED, `/cancel` = {QUOTED,HELD,CONFIRMED}→CANCELLED. **Deviates from master plan: no** — it RESOLVES the §2.4/§5 underspecification (§5's hold acts on the quote; §2.4's machine needs a booking in HELD; two enums share the token HELD) and reconciles the §2.3-vs-§6.2 discrepancy (**§6.2/§5 authoritative**: two calls, create then confirm; §2.3's single arrow is diagram compression). Also records: **actor-agnostic confirm** (booking-service stamps the caller's `actor`; the "user click only" gate lives UPSTREAM in agent-service §6.3.2, since the booking API is the same public API for UI and agent §2.2); **first-write-wins idempotency** (replay returns the original, 200 vs 201; divergent payload ignored; race-safe via pre-SELECT + `23505` + StateConflict re-check); **quote_status** uses guarded transitions, not the State pattern (§4.4 scopes State to bookings).
