## Two latent PRODUCTION provider findings the harness surfaced (drive-the-real-path paying off — flagged, NOT fixed here)
- **(a) Gemini — the PRIMARY provider — rejects the tool SCHEMA:** `shipmentJsonSchema` uses `exclusiveMinimum`, which Gemini's function-declaration dialect does not accept → **400, non-retryable → no fallback**. Any real turn using the full `TOOLS` set against Gemini-primary would 400. D14 never caught it because its tests mock HTTP (a mock fakes the 200; it never lets the provider reject the REQUEST).
- **(b) Groq server-side-validates tool calls** and 400s `tool_use_failed` on schema-invalid promptless args, crashing the turn BEFORE the loop's Zod-retry engages. Consequence: the `v0-none` recordings were captured from the Groq **FALLBACK**, and the 3 extraction error-cases are these real 400s replayed as FAILs.
- Both are for **L5/debugger** to fix (the harness makes zero production changes): drop/translate `exclusiveMinimum` so the primary accepts the schema; make invalid tool args survive Groq's validation into the loop's Zod-retry.

## Learned
- LEARNING.md appended ×3: (1) **Evals & Testing** — driving the REAL code path (import the production loop/tools/router; replay real provider bytes) surfaced two provider-fatal SCHEMA bugs D14's mocked-HTTP tests structurally could not (Gemini rejects `exclusiveMinimum`; Groq 400s `tool_use_failed`) — a mock fakes the response, so it can't catch the dependency rejecting your REQUEST.
