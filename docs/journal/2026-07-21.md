L3 mints the confirmation token → reuses it as BOTH the create `Idempotency-Key` AND the gate credential, and fills `bookingId` from the create result.

## The decisions (recorded in ADR-0009 — §6.3.2 for the table + 10-min TTL)
- **Token binding (A):** opaque server-side token keys a stored server-authoritative proposal; redeem carries ONLY the token → a token authorizes THIS booking, not "a booking," and can't be replayed against a different proposal.
- **create/confirm idempotency ASYMMETRY + reconcile-by-read (D — the load-bearing correctness decision):** create is idempotent on `Idempotency-Key=token` (first-write-wins); confirm is NOT (CONFIRMED→CONFIRMED = 409 ILLEGAL_TRANSITION, proven by booking's `idempotency.it.test` line 95). Recovery is reconcile-by-READ (GET booking; CONFIRMED ⇒ success/ack-lost; HELD ⇒ retry; bounded 2), never a blind re-POST reading 409 as failure. This is WHY the single-use claim (B) is mandatory, not belt-and-suspenders.
- **Token unguessability is DO-NOW (F), not deferred:** auth is a Non-Goal → possession of the token IS the authorization; 256-bit crypto-random, masked out of access logs, kept out of `booking_events` metadata (the non-secret `id` goes there).
- **loop-stays-pure / separate-gate-service (Flag 5):** executor structurally unreachable from the loop (C — static import-graph test, hardened vs dynamic import/require).
- **Gate endpoints are UI-INTERNAL (Flag 7):** agent publishes NO OpenAPI contract — same-origin L5 chat UI, not external clients. Deliberate; recorded so it's tracked.
- **Known limitations RECORDED (auth is a Non-Goal per ADR-0005):** tokens not tenant-scoped; actor caller-trusted / unauthenticated. Acceptable-for-now.
