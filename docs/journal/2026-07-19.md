# Journal — 2026-07-19

**Phase/Layer:** Phase 1 / L4 booking half — **PR-A** (rates-contract precondition). A single-concern precondition that unblocks the deferred booking half of L4 (ADR-0004). NOT the booking UI itself — this is the contract seam the client (later the agent) needs before it can assemble a booking quote from a rates quote.
**Session goal:** Add `lane_id` to the rates `QuoteResponse` so a consumer can forward the `(rate_card_id, lane_id)` pair verbatim into booking-service's `POST /quotes` (which REQUIRES `lane_id`) — without the client inventing rates-owned data (§2.2). Ship it additive and provably non-breaking.

## The seam (why PR-A exists)
- booking-service `CreateQuoteRequest` requires `lane_id`, but the rates `QuoteResponse` / `RateCardView` exposed no `lane_id`. So the client could not build a booking quote from a rates quote without synthesizing or mapping rates-owned data — a §2.2 violation (client owns no business logic; rates owns lanes).
- architecture-guardian ruled the boundary-preserving fix is for **rates** to emit the pair (cross-service data flows through REST contracts only, §2.2), NOT for the client to invent or map it. This is additive completion of the existing contract, not a new decision — the decision was already made and recorded in **ADR-0004** (defer the booking half; client invents nothing; "stub a fake path" already rejected). No new ADR.
